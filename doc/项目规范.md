## 使用说明
- 本文档不是“所有项目统一使用同一套技术栈”的定义。
- 本文档用于按项目类型区分开发规范，AI IDE 必须先判断当前项目属于哪一类，再选择对应规范执行。
- 通用规范适用于所有项目。
- 语言/技术栈专属规范只适用于对应类型项目，禁止把 Node 项目规范直接套用到 Python 项目，反之亦然。

---

## 一、通用项目规范

### 1. API 实现规范
- 使用清晰、稳定的 HTTP API 设计。
- 默认使用 JSON 作为数据交换格式。
- 使用 HTTP 状态码表达请求结果。
- 需要鉴权时，应使用项目约定的认证方式。
- 需要接口文档时，应使用项目对应生态的标准文档方案。

### 2. 代码规范
- 创建的类、函数都增加 `/** 注释 */`。
- 不使用带有 `@deprecated` 声明的函数、类或 API。
- 优先保持单一职责，避免把多个业务目标塞进同一个模块。
- 修改代码前先判断当前需求属于哪个节点、哪个模块、哪个责任边界。

### 3. 工程协作规范
- 先读取项目自身技术栈，再选择对应规范，不得反向假设。
- 如果当前项目技术栈与本文档中的某个分类不一致，应只沿用通用规范，不得硬套不匹配的技术细节。
- 若项目已存在明确目录结构、依赖管理方式、测试框架或部署方式，应优先遵循项目现状。

---

## 二、Node.js / TypeScript 项目规范

### 1. 适用范围
- 适用于基于 Node.js 运行时的项目。
- 适用于 TypeScript / JavaScript 服务端项目。
- 适用于 NestJS、Express、Fastify 等 Node 生态项目。

### 2. 推荐技术栈
- 开发语言：TypeScript
- 包管理器：pnpm

### 3. 开发框架
- NestJS：基于 TypeScript 的 Node.js 框架
- NestJS Fastify：NestJS 的 HTTP 引擎

### 4. 数据库
- PostgreSQL：关系型数据库

### 5. 配置
- YAML：配置文件格式

### 6. 缓存
- ioRedis：Redis 缓存客户端

### 7. 日志
- Winston：Node.js 日志库

### 8. 身份验证
- NestJS Passport：身份验证集成方案

### 9. 静态资源
- NestJS ServeStatic：静态资源服务

### 10. 测试
- Jest：Node.js / TypeScript 项目测试框架

### 11. 流水线
- GitLab CI/CD：持续集成与部署

### 12. 云服务
- AWS S3：静态资源存储
- AWS EC2：云主机
- AWS Bill：账单管理

### 13. 微服务
- NestJS Microservices：微服务框架
- gRPC：高性能 RPC 通信

### 14. 容器化
- Docker：容器化
- Docker Compose：多容器编排

### 15. 部署
- Kubernetes：容器编排
- Helm：Kubernetes 包管理

### 16. AI 服务
- MCP：用于 AI IDE / Agent 协作能力集成

### 17. Node 项目附加要求
- 优先使用 TypeScript。
- 优先使用项目现有的包管理器；若项目明确使用 pnpm，则不得切换为 npm 或 yarn。
- 优先复用现有框架约定，例如 NestJS 的 module / service / controller 分层。

## 三、Python 项目规范

### 1. 适用范围
- 适用于基于 Python 运行时的项目。
- 适用于 Flask、FastAPI、Django、脚本型后端或数据处理项目。

### 2. Python 项目默认原则
- 包管理器、虚拟环境、框架、测试框架必须以项目当前实际配置为准。
- 未确认项目实际依赖管理方式前，不得假设使用 pnpm、Node、NestJS、Jest、Passport 等 Node 生态能力。
- 未确认项目实际 Web 框架前，不得假设使用 REST 控制器分层结构与 Node 风格模块结构。

### 3. Python 项目常见实现方向
- 开发语言：Python
- 依赖管理：以项目现有配置为准，例如 `uv`
- 测试框架：以项目现有配置为准，例如 `pytest`、`unittest`
- Web 框架：以项目现有配置为准，例如 `FastAPI`

### 4. Python 项目附加要求
- 优先遵循项目已存在的虚拟环境、依赖锁定和入口脚本。
- 优先遵循项目既有目录结构，不得套用 Node 项目的 `controller/service/module` 结构。
- 若项目没有明确工程约定，应先识别项目现有文件和依赖，再决定实现方式。

---

## 四、AI IDE 执行规则
- AI IDE 在读取本文档后，必须先判断当前项目属于：
  - 通用项目
  - Node.js / TypeScript 项目
  - Python 项目
- 判断依据应来自：
  - 项目目录结构
  - 锁文件
  - 配置文件
  - 依赖清单
  - 现有代码
- 若当前项目是 Node.js / TypeScript 项目，则同时遵守：
  - 通用项目规范
  - Node.js / TypeScript 项目规范
- 若当前项目是 Python 项目，则同时遵守：
  - 通用项目规范
  - Python 项目规范
- 若项目类型无法确认，先按通用规范执行，并优先补充识别项目技术栈，不得直接套用某一类专属规范。
