1.0 读取项目规范.md文档来了解项目使用哪些技术和API规范，当有疑问时使用context7工具查询对应的技术文档
- 不需要执行eslint，只需要执行tsc
- 读取项目结构.md文档，了解项目的目录结构
- 当前项目使用pnpm作为包管理器，typescript作为编程语言

1.1 先通过 ai-spec-develop 的 MCP 工具完成协作初始化：
- 调用 `project_tool(action="register_client")` 注册当前 AI IDE 会话信息
- 调用 `project_tool(action="confirm_execution_scope")` 明确本次执行的 `projectId` 和 `startNodeId`
- 调用 `task_tool(action="get_collaboration_context")` 读取当前局部范围内其他 AI 会话的参与者、执行计划、最近交接记录

1.2 在需求解析开始前，必须先确认目标项目类型和适用规范：
- AI 不得默认把当前仓库中的某一类技术规范直接套用到所有目标项目
- 在解析需求前，必须先确认当前目标项目属于哪一类，例如：
  - Node.js / TypeScript 项目
  - Python 项目
  - 其他类型项目
- 必须确认当前任务应套用哪一套项目规范，再继续拆解需求和设计实现
- 如果项目类型尚不明确，应先向用户确认，再进入需求拆解或编码阶段
- 若用户尚未明确项目类型，AI 只能先按通用工程原则做需求理解，不能直接套用 Node、Python 或其他专属规范

2.必须先获取项目和任务上下文，禁止跳过：
- 调用 MCP 工具获取目标项目列表，定位“博客管理系统”项目
- 继续调用 MCP 工具获取项目完整信息、工作流节点、节点依赖、执行历史
- 调用 `task_tool(action="get_pending_tasks")` 获取当前局部范围内的待办节点

2.1 当需求描述不充分时，禁止直接实现最简版本：
- 如果当前节点只描述了功能目标，但没有明确接口、页面组成、数据结构、状态流转、依赖模块或验收标准，则视为“需求信息不足”
- 需求信息不足时，AI 不能直接开始编码，也不能只实现一个最小可运行功能来交差
- 必须先扩展当前节点的子级工作流，再按子级节点逐步实现
- 子级节点至少应覆盖：接口设计、服务实现、数据结构、页面组件、状态管理、联调验证、异常处理

2.2 扩展子节点时的判断规则：
- 先判断当前节点是否已经具备“可直接编码”的完整边界
- 如果无法明确知道“要改哪些文件、暴露哪些接口、页面由哪些区域组成、依赖哪些前后端能力”，则必须先拆分子节点
- 拆分后的子节点应保持单一职责，并能够分别声明执行范围和执行意图
- 不能把一个抽象节点直接实现成单文件、单接口或单页面的简化版本，除非节点需求已经明确限定为该颗粒度

2.2.1 扩展子节点时必须按两层视角输出：
- `requirement` 字段只用于业务视角描述
- 业务视角必须说明：该节点完成什么业务目标、服务哪些用户、交付什么业务结果、与上下游业务有什么关系
- `prompt` 字段只用于技术视角描述
- 技术视角必须说明：涉及哪些模块、接口、数据结构、页面、状态流转、实现约束和验证方式
- 禁止把技术实现细节写进 `requirement`
- 禁止把纯业务描述写进 `prompt`

2.2.2 同级子任务必须维护先后顺序：
- 当 AI 扩展同一父节点下的多个子节点时，必须显式维护同级排序
- 应通过 `sortOrder` 表达这些同级任务的执行顺序，数值越小越靠前
- 如果多个子任务存在明确先后关系，不能只依赖创建时间或默认顺序
- 同级子任务排序应优先体现业务实现顺序、依赖顺序和联调顺序
- 禁止省略 `sortOrder`，也禁止把所有同级节点都写成相同顺序值

2.2.3 依赖关系必须显式写入 `dependencies`：
- `parentNodeId` 只表示层级归属，不表示执行依赖
- 只要某个子节点必须等待另一个节点完成后才能执行，就必须把该关系写入 `dependencies`
- 不能仅通过父子结构暗示依赖关系，必须显式声明依赖节点 ID 列表
- 如果多个子节点之间存在前后置关系，应同时维护：
  - `dependencies`：表达执行阻塞关系
  - `sortOrder`：表达同级推荐执行顺序
- 没有真实执行依赖时，`dependencies` 应为空数组，而不是滥用父节点或根节点

2.3 扩展子节点的执行规则（硬性约束）：
- 当 AI 判断当前节点需要扩展子级时，必须先补全工作流子节点，再进入编码阶段
- 补全子节点时，应通过 MCP 的节点工具同步节点结构，例如调用 `node_tool(action="sync_workflow_nodes")` 或项目中等效的工作流同步工具
- 同步子节点时，必须同时写入业务视角 `requirement`、技术视角 `prompt`、显式依赖 `dependencies` 和同级排序 `sortOrder`
- 在子节点尚未写回工作流之前，禁止直接开始实现父节点代码
- 新增子节点后，应重新确认依赖关系、执行范围和任务分配，再按子节点逐个执行
- 父节点在其子节点未完成前，不应被直接标记为完成

3.严格遵守依赖等待规则，禁止抢跑联调：
- 在执行任一节点前，必须先调用 `task_tool(action="check_sync_status")`
- 如果返回 `BLOCKED`，说明前置节点尚未完成，此时禁止修改代码、禁止假设接口已存在、禁止自行创建替代接口
- 如果返回 `BLOCKED`，当前 AI 必须进入等待态，并定时调用 `task_tool(action="check_sync_status")` 或 `tasks/get` 轮询依赖节点状态，直到依赖节点完成
- 前端节点依赖后端节点时，前端 AI 必须等待后端 AI 完成接口实现并提交完成，不能提前联调
- 只有在依赖节点 READY 后，才能调用 `task_tool(action="declare_intent")` 声明自己的执行计划并开始修改文件

4.多个 AI 必须基于 MCP 声明协作，避免执行冲突：
- 修改代码前必须调用 `task_tool(action="declare_intent")`
- `intent` 里必须明确写清楚：准备修改哪个模块、哪个文件、哪个接口或哪个页面
- 必须先参考 `get_collaboration_context` 和 `get_node_history` 中其他 AI 的声明和交接记录，再继续实现
- 如果某节点已被其他会话声明执行，则当前 AI 不得重复实现该节点
- `declare_intent` 成功后，表示当前节点已经进入执行中；在释放执行权前，AI 不能无状态退出或跳过任务回写

5.前后端联调时，接口路径和契约只能来自后端真实实现，禁止猜测：
- 前端 AI 发起 API 请求前，必须先检查后端节点是否已完成
- 前后端接口必须遵循标准 HTTP 规范设计，优先使用清晰的 REST 风格路径和语义化方法，例如 `GET`、`POST`、`PATCH`、`DELETE`
- 前端使用的请求路径、请求方法、路径参数、查询参数、请求体结构、响应结构，必须来自后端已完成代码、项目节点需求、节点执行历史或 MCP 协作上下文
- 禁止前端自行猜测 `/api/xxx`、字段名、状态码、返回结构，也禁止自行设计 `POST + action` 这类内部约定接口来绕过后端契约
- 如果后端尚未完成或接口契约不明确，前端 AI 必须等待，或读取后端 AI 的执行历史后再接入

5.1 当需要扩展需求边界时，必须优先参考示例项目或参考模块：
- 如果当前项目中存在示例项目、参考项目、相似模块或历史实现，AI 必须优先参考其模块划分、接口设计、页面结构和联调流程
- 示例项目用于帮助 AI 判断“应该拆到什么颗粒度”，以及“完整功能通常包含哪些子模块”
- 参考重点是工程结构、模块边界、接口规范、交互分层，而不是直接复制业务内容
- 若提示词或项目上下文中指定了示例项目路径，AI 应先读取该示例项目，再决定当前节点是否需要扩展子级
- 如果没有示例项目，则优先参考当前仓库中已存在的相似功能模块，其次再基于通用工程经验补全子节点

5.2 示例项目的使用规则：
- 示例项目只用于确定功能拆分方式、文件组织方式、接口风格和页面结构
- 不允许照搬示例项目中的业务名称、文案、数据内容或与当前需求无关的实现
- AI 应明确区分“参考结构”与“实现当前业务”
- 若参考后发现当前节点缺失必要子任务，应先补全子节点，再继续编码

6.完成节点时必须形成闭环：
- 节点实现完成并验证后，调用 `task_tool(action="submit_completion")` 提交输出摘要
- `outputSummary` 必须写明：完成了哪些功能、涉及哪些文件、暴露了哪些标准 HTTP 接口或消费了哪些标准 HTTP 接口
- 节点完成后会自动更新为 `completed`

6.1 任务状态回写是硬性约束，禁止只改代码不回写：
- 任何 AI 在一次执行中，只要已经修改了代码或明确得出了任务结论，就必须在结束本轮前通过 MCP 回写任务状态
- 如果功能已完成：必须立即调用 `task_tool(action="submit_completion")`
- 如果因为依赖未完成、信息不足、接口未就绪而无法继续：必须调用节点工具把状态更新为 `review_needed` 或项目约定的阻塞状态，并在执行摘要中说明阻塞原因
- 如果实现失败或确认当前方案不可行：必须调用节点工具把状态更新为 `failed`，并写明失败原因、影响范围和下一步建议
- 禁止出现“代码已经提交或已经完成分析，但节点仍保持 pending / in_progress 且没有摘要”的情况

6.2 回写状态的时机要求：
- AI 不得在完成实现后先结束对话、先等待用户、或先开始下一个任务，再补写 MCP 状态
- 正确顺序必须是：完成实现/得出结论 → 立即调用 MCP 回写状态 → 再结束当前轮次
- 如果一个节点已经达到可交付状态，但没有调用 MCP 回写，则视为任务未完成
- 如果一个节点进入执行中后长时间没有新的回写动作，应优先检查是否遗漏了 `submit_completion` 或节点状态更新

6.3 无法完成时也必须回写：
- “做不完”“被阻塞”“需要确认”“测试环境不足”都不是不回写状态的理由
- 遇到这些情况时，AI 必须把节点状态更新为可监控状态，并在摘要中清楚记录当前结论
- 只有完成了状态回写，这次执行才算真正结束

7.执行目标：
- 优先按照工作流依赖顺序推进后端功能，再推进依赖这些接口的前端功能
- 所有 AI 必须通过 MCP 工具共享执行范围、执行计划、依赖状态和完成摘要
- 任务必须完成，但不能通过绕过依赖、伪造接口、重复实现来完成
- 当节点需求抽象度过高时，优先扩展工作流子节点，而不是直接实现最简功能


### 多Agent协作协议
- 🔄 Agent 协作协议 v1.2 (无推送适配版)
#### Rule 1 — 长轮询探测 (Long-Polling Check)
在处理任何任务节点前，必须调用 task_tool(action="wait_and_check_status", timeout=30)，传入当前 nodeId。

若服务端返回 READY： 包含前置节点的 I/O Schema 契约，直接进入下一步。

若服务端返回 TIMEOUT_BLOCKED： 意味着经过了最长等待时间，前置任务仍未完成。

强制动作： AI 必须立即停止执行，输出：“⏸ 前置任务仍在进行中，为了节省资源，我已暂停。请在依赖完成后回复‘继续’。” 绝对禁止 AI 自行发起下一次循环调用。 必须将控制权交还给人类或外部触发器。

#### Rule 2 — 独占挂号与防抖 (Debounced Lock)
确认 READY 后，必须调用 task_tool(action="acquire_lock") 声明意图。

注意： 由于网络延迟或并发，可能会有多个 AI 实例几乎同时通过了 Rule 1。服务端在这一步必须执行严格的原子化加锁（如 Redis 的 SETNX）。

只有拿到 LOCK_SUCCESS 才能开始写代码。拿到 LOCK_FAILED 说明被其他 Agent 抢占，必须退回 Rule 1。

#### Rule 3 — 契约驱动执行 (Contract-Driven Execution)
在锁定状态下进行最小化模块开发。严格遵循 Rule 1 返回的数据契约，严禁修改任何不属于当前 nodeId 职责范围的文件。如果发现契约有误，直接执行 Rule 4 的失败分支，由人类介入调整架构图。

#### Rule 4 — 状态提交与依赖释放 (Submit & Release)
执行完毕且本地测试通过后，调用 task_tool(action="submit_completion")。

提交通过后，NestJS 服务端会在内部触发事件，将状态机中该节点标记为 COMPLETED。

此时，如果有其他 Agent 的 Rule 1 长轮询请求正被挂在 NestJS 后端，它们会被立刻唤醒并收到 READY 响应，实现无缝衔接。

---
**违反上述任意规则将导致多 Agent 协作状态不一致，可能引发并发冲突或数据丢失。**
